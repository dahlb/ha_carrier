1"""Retry policies and shared escalation state for Carrier API operations. 

2 

3Carrier can return short-lived 401s and transient transport failures during 

4service outages, so the integration does not treat the first failure as a hard 

5reauth or permanent outage. Each coordinator owns one `ResiliencyState` shared 

6by refresh and write API calls. Websocket failures request a refresh instead of 

7changing counters directly, so the refresh path remains the owner of outage 

8accounting. A later successful API operation clears stale counters unless the 

9caller is intentionally doing cycle-level accounting. 

10 

11`async_call_with_retry` is the shared helper for bounded API calls. The websocket 

12loop reuses `RetryPolicy` and `compute_backoff_delay` directly because it manages 

13its own long-running listener and reconnect cycle. 

14"""

29@dataclass(frozen=True) 

30class RetryPolicy: 

31 """Declarative retry behavior for one logical API operation type. 

32 

33 The policy describes how one call retries after a classified failure. It 

34 does not own escalation counters; those live in `ResiliencyState` so 

35 refreshes and writes contribute to the same outage window. 

36 

37 Attributes: 

38 name: Short identifier used in log messages. 

39 max_attempts: Total attempts including the first call. None = unbounded. 

40 base_delay: Initial backoff delay in seconds (before jitter). 

41 max_delay: Maximum backoff delay in seconds (after exponentiation). 

42 jitter_fraction: Symmetric jitter as a fraction of the computed delay. 

43 retry_on_unauthorized: When True, a 401 inside the threshold is retried 

44 in-place with backoff; when False, non-escalated unauthorized 

45 errors are re-raised to the caller and only threshold escalation 

46 raises CarrierUnauthorizedError. 

47 """ 

48 

49 name: str 

50 max_attempts: int | None 

51 base_delay: float 

52 max_delay: float 

53 jitter_fraction: float 

54 retry_on_unauthorized: bool

57@dataclass 

58class ResiliencyState: 

59 """Cross-call escalation counters shared across coordinator surfaces. 

60 

61 Unauthorized and transient failures are tracked separately because they 

62 escalate to different Home Assistant behavior: persistent 401s trigger 

63 reauthentication, while persistent transport failures surface as retryable 

64 update/write failures. A successful normal operation resets both counters. 

65 Callers that need cycle-level accounting, such as per-system energy refresh, 

66 can defer success resets and record the cycle result explicitly. 

67 

68 Attributes: 

69 unauthorized_threshold: Consecutive 401 count that triggers escalation. 

70 transient_threshold: Consecutive transient failure count that triggers 

71 escalation. 

72 consecutive_unauthorized: Current count of consecutive 401-style 

73 failures in the shared outage window. 

74 consecutive_transient: Current count of consecutive transient transport 

75 failures in the shared outage window. 

76 unauthorized_outage_logged: Whether the current unauthorized outage has 

77 already emitted its first log message. 

78 unauthorized_escalated_logged: Whether the current unauthorized outage 

79 has already emitted its escalation log message. 

80 transient_outage_logged: Whether the current transient outage has 

81 already emitted its first log message. 

82 transient_escalated_logged: Whether the current transient outage has 

83 already emitted its escalation log message. 

84 """

177def compute_backoff_delay(policy: RetryPolicy, attempt: int) -> float: 

178 """Return the next backoff delay for a given policy and 0-indexed attempt. 

179 

180 Computes `base_delay * 2 ** attempt`, clamps to `max_delay`, and applies 

181 symmetric jitter of `±jitter_fraction * delay`. 

182 

183 Args: 

184 policy: Retry policy whose timing parameters drive the calculation. 

185 attempt: Zero-based attempt number (0 = first retry after first failure). 

186 

187 Returns: 

188 float: Non-negative delay in seconds. 

189 """ 

190 if policy.base_delay > 0 and policy.max_delay > policy.base_delay: 

191 max_exponent = ceil(log2(policy.max_delay / policy.base_delay)) 

192 else: 

193 max_exponent = 0 

194 exponent = min(max(attempt, 0), max_exponent) 

195 raw = policy.base_delay * (2**exponent) 

196 capped = min(raw, policy.max_delay) 

197 capped = max(0.0, capped) 

198 if policy.jitter_fraction <= 0: 

199 return capped 

200 jitter = capped * policy.jitter_fraction 

201 return max(0.0, capped + random.uniform(-jitter, jitter))

214 """Run `operation` with classification-driven retry and shared escalation state. 

215 

216 The helper classifies failures before deciding whether to retry, escalate, 

217 or re-raise. Unauthorized failures optionally update the shared auth counter; 

218 once the threshold is crossed, `CarrierUnauthorizedError` is raised so the 

219 coordinator can trigger Home Assistant reauth. Non-escalated 401s either 

220 retry in-place when the policy allows it or propagate to the caller for 

221 cycle-level handling. 

222 

223 Transient transport failures update the shared transient counter. They retry 

224 with exponential backoff until the policy's attempt limit is reached or the 

225 shared transient threshold escalates, in which case the original error is 

226 raised. 

227 

228 A successful call normally resets all shared resiliency state. Callers set 

229 `reset_state_on_success=False` only when one logical operation is made from 

230 multiple helper calls and will record/reset state after the full cycle. Other

232 and `SystemExit` are always re-raised.
